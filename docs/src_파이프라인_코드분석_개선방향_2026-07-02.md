# src 파이프라인 코드 분석 및 개선방향 (2026-07-02)

> `src/`의 넘버링 파이프라인(00~30)을 정밀 검토한 결과와 개선방향입니다.
> 전반부(00~15: 데이터·HSI 신호·flex 백테스트)와 후반부(16~30: main_v2/v2b 백테스트·시각화·v3 설계)를 나눠 분석했습니다.

## 1. 파이프라인 전체 구조

넘버링 스크립트는 **서로를 import하지 않고, 오직 CSV 파일로만 데이터를 주고받는** 구조입니다. 각 스크립트가 CSV를 다시 읽고 날짜를 다시 파싱하며, 파일명이 사실상 스크립트 간 계약(contract) 역할을 합니다.

크게 두 개의 하위 파이프라인으로 갈라집니다.

- **00~09 (원본 다자산 계열):** `data/raw/korea_etf.csv` → 정제 → 이벤트 등급 분류(04) → HSI 상태 라벨(05) → 시각화/구간분할/해석(06~09)
- **10~15 (flex 3종 ETF 계열):** yfinance로 3개 ETF(069500/114260/153130) 로드 → HSI 입력신호(10_build) → 월말 상태(12) → 수익률 정합(13) → EW+HSI 예비 백테스트(14~15)
- **16~30 (main_v2 계열):** 데이터팀의 3분류(buy/watch/caution)를 5상태로 재분류(16) → overlay 백테스트 v2(17)/v2b(19) → 성과평가(18/20/21) → 시각화(24~28) → 인덱스/계획/v3 설계(22/23/29/30)

**실제로 백테스트를 "실행"하는 스크립트는 17(v2)과 19(v2b) 둘뿐**이며, 18·20·21은 평가, 24~28은 시각화, 22·23·29·30은 인벤토리/계획/설계 산출물입니다. 특히 **30(v3)은 백테스트를 실행하지 않고 설계표만 생성**합니다(모든 행이 `status="planned"`).

## 2. 잘 되어 있는 점 (유지할 강점)

1. **경로가 모두 상대 기준(`Path(__file__).resolve().parents[1]`)** — 하드코딩된 절대경로가 없어 이식성이 좋습니다.
2. **Look-ahead(미래참조) 방어가 핵심 지점에서 정확합니다.**
   - 04는 롤링 분위수 임계값에 `.shift(1)` 적용, 05는 252일 gap 분위수에 `.shift(1)`.
   - 13과 17/19 백테스트는 월말 t 시점 HSI 상태를 **다음 달(t+1) 수익률**에 적용(`shift(-1)`)해 동일월 누출을 막습니다. 이 부분은 도식/주석으로도 명시돼 있습니다.
3. **가중치 합 검증(`np.allclose`)·컬럼 존재 검증** 등 전제조건 `raise` 가드가 곳곳에 있습니다.
4. **NaN 처리가 명시적입니다** — `errors="coerce"`, 분모 0 가드(Sharpe/Calmar/Sortino), rolling window 부족 시 `insufficient_data` 라벨링.
5. **함수 분해가 대체로 양호**하고 모듈 docstring이 입력/출력을 설명합니다. `utf-8-sig` 인코딩도 일관적입니다.

## 3. 주요 문제점 (우선순위순)

### 🔴 P1. 심각 — 반드시 개선 권장

**(1) 공통 모듈 부재로 인한 대규모 코드 중복**
경로 블록, 한글 폰트 설정, CSV 로더, 백테스트 엔진, 성과지표 함수가 파일마다 복사·붙여넣기되어 있습니다.

- `read_csv_with_date`: 최소 9개 파일에 복제(11/12/13/16/17/19/25/26/27), 그마저 strict/lenient 두 변종이 혼재.
- `set_korean_font()` / `configure_korean_font`: 06/09/24/25/26/27/28/10_visualize에 동일 코드 반복.
- **백테스트 엔진 전체가 17≈19에 복제**(`align_state_with_next_returns`, `build_backtest_for_method`, `calculate_turnover`, `make_turnover_summary`가 거의 동일). v2/v2b 차이는 `conflict` 상태 가중치와 전략 이름 문자열뿐.
- **성과지표 함수가 18/20/21에 복제** — 그런데 **20의 복사본은 이미 원본과 어긋났습니다**(Sortino 누락, `months==0` 가드 누락). 이것이 중복의 위험을 그대로 보여주는 실제 사례입니다.

→ 가중치 하나만 바꿔도 여러 파일을 동시에 고쳐야 하며, 이미 일부는 값이 달라져 있습니다.

**(2) 파일 넘버링 충돌 — `10_` 3개**
`10_build_hsi_signal_inputs.py`, `10_visualize_hsi_hourglass_cone.py`, `10_build_presentation_state5_tables.py`가 같은 번호를 공유합니다. 세 파일의 상호 실행 순서가 불명확하고(`10_build`는 12가 소비, 나머지 둘은 05 산출물 의존), `10_*.py` 글롭 실행 시 순서가 비결정적입니다.

**(3) main_v2 분류 단계의 in-sample 미래참조**
16의 intensity/accident 임계값이 **전체 기간(미래 포함) 분위수**로 계산됩니다(`HIGH_INTENSITY_QUANTILE=0.75`, `ACCIDENT_DIRECTION_QUANTILE=0.85`를 전체 표본에 적용). 백테스트 alignment는 깨끗하지만, **상태 분류 자체가 전체 표본 통계에 의존**하므로 엄밀한 out-of-sample 해석에는 약한 누출이 있습니다. design/validation 분리(07) 취지와도 상충합니다.

### 🟠 P2. 중요 — 신뢰도·재현성 관련

**(4) 재현성: yfinance 라이브 로드가 기본값**
`10_build`의 `load_price_data`가 `source="yfinance"`, `end=None`(오늘까지)로 기본 동작합니다. 실행할 때마다 가격이 갱신되어 **10~15 산출물이 재현되지 않습니다.** 또한 `source="csv"` 경로는 `korea_etf.csv`와 다른 파일명 스킴(`{ticker}.csv`)을 써서 00~09 계열과도 불일치.

**(5) 검증 결과가 실제 검증이 아님 (가짜 OK 플래그)**
17/19가 쓰는 `alignment_flag = "OK"`는 **하드코딩된 리터럴**입니다. `missing_next_return_cells`를 계산은 하지만 어디서도 분기하지 않아, 실제 정합성과 무관하게 항상 "OK"를 기록합니다. 날짜가 연속 월말인지 검증하지 않아, `monthly_returns`에 공백 월이 있으면 `shift(-1)`이 조용히 어긋날 수 있습니다(위치 기반 shift, 달력 기반 아님).

**(6) 거래비용이 코드에 반영되지 않음**
v2/v2b 수익률은 **거래비용 미반영 총수익(gross)**입니다. 거래비용은 30(v3)에 설계로만 존재하고 실행되지 않습니다. 회의록에서 확정한 `월별 거래비용 = Turnover × 비용률` 로직이 아직 백테스트에 없습니다.

### 🟡 P3. 개선 권장 — 유지보수성

**(7) 매직넘버 산재:** 04의 `LOOKBACK_DAYS=60/SMALL_Q=0.60/LARGE_Q=0.90`, 05 점수 컷오프, 14 overlay 가중치(0.50/0.15/0.425), 16 분류 임계값, design/validation 경계일(07·10_state5에 각각 하드코딩) 등.

**(8) 테스트 전무:** `test_*.py`·pytest·`assert` 없음. 검증은 stdout `[완료]` 프린트와 리포트 CSV뿐이며, 중복 날짜·결측 셀을 "기록"만 하고 실패시키지 않습니다.

**(9) 타입힌트·docstring 부분적:** `10_build` 핵심 함수(`compute_hsi` 등)와 24~29 plot/main 함수에 타입힌트·docstring 누락.

**(10) 상태 분류 체계 이원화:** 05는 8상태 한글 라벨, 10_state5·16은 5상태 영문 키를 공유 정의 없이 병존. 루트 경로 별칭도 `PROJECT_DIR`/`ROOT_DIR`/`PROJECT_ROOT`로 제각각.

**(11) 기타:** `10_build`의 전역 `warnings.filterwarnings("ignore")`(deprecation 경고까지 은폐), 160줄짜리 `__main__` 블록(script-style), `== True`/`== False` 비교(22/29), 미사용 import(00의 numpy).

## 4. 개선방향 (구체적 제안)

### 4.1 공통 모듈 도입 — `src/common/` 패키지 (최우선)

중복을 한 곳으로 모읍니다. 예시 구조:

```
src/common/
  __init__.py
  paths.py       # PROJECT_ROOT, TABLE_DIR, FIGURE_DIR, DOCS_DIR, DATA_DIR
  io_utils.py    # read_csv_with_date(strict/lenient 통합), save_table()
  viz.py         # set_korean_font(), STRATEGY_LABEL_MAP
  backtest.py    # align_state_with_next_returns(), build_backtest(), calculate_turnover()
  metrics.py     # calculate_performance_metrics() (단일 정본)
  config.py      # ASSETS, BENCHMARK_TICKER, 날짜 경계, 임계값 상수
```

- v2와 v2b의 유일한 차이(=`conflict` 가중치)는 **allocation_rule 테이블을 인자로 주입**해 하나의 `build_backtest(rule_table, strategy_name)`로 통합. 17/19의 엔진 중복이 사라집니다.
- 성과지표는 `metrics.py` 단일 함수로 통일 → 20의 "어긋난 복사본" 문제 근본 해결.
- 각 넘버링 스크립트는 `from common import ...`만 하면 되어 라인 수가 크게 줄고 드리프트가 사라집니다.

### 4.2 파일 넘버링 정리

`10_` 3개를 실행 순서에 맞춰 재배치:
- `10_build_hsi_signal_inputs.py` → 유지(10)
- `10_visualize_hsi_hourglass_cone.py` → 예: `10b_` 또는 시각화 계열 번호로
- `10_build_presentation_state5_tables.py` → 예: `10c_` 또는 별도 `presentation/`

또는 스크립트 상단에 `# depends: 05_..., produces: hsi_summary_*.csv` 형태의 의존성 주석 표준을 도입.

### 4.3 재현성 확보

- `10_build` 기본값을 `source="csv"`로 바꾸고, 원천 CSV(`data/raw/`)를 단일 진실원(source of truth)으로 고정. yfinance는 명시적 `--refresh` 플래그로만.
- 다운로드 시점의 **데이터 스냅샷과 기간(start/end)을 메타 파일로 저장**해 실행 재현 가능하게.
- `config.py`에 `RANDOM_SEED` 인프라를 두어(현재는 무작위성 없지만) 향후 Grid Search 샘플링/부트스트랩 확장 대비.

### 4.4 검증을 "진짜" 검증으로

- `alignment_flag`를 리터럴이 아니라 **계산 결과로**: `flag = "OK" if missing_next_return_cells == 0 and dates_are_contiguous else "CHECK"`.
- 월말 날짜 연속성 검증 함수 추가(달력 기반 gap 탐지).
- 경량 테스트 도입: `tests/test_metrics.py`(알려진 입력→기대 CAGR/MDD), `tests/test_backtest_alignment.py`(t 상태가 t+1 수익률에 매핑되는지 assert), `tests/test_weights_sum.py`. pytest 1개 파일부터 시작해도 회귀 방지 효과가 큽니다.

### 4.5 거래비용을 백테스트에 실제 반영

30 설계표의 로직을 17/19(통합 후 `backtest.py`)에 구현:
```python
monthly_cost = turnover * cost_rate           # cost_rate ∈ {0, 5, 10, 20}bp
net_return   = gross_return - monthly_cost
```
비용률별 컬럼(`ret_after_cost_0bp/5bp/10bp/20bp`)을 산출해 v3 최종 판단표로 바로 연결.

### 4.6 분류 단계 미래참조 제거

16의 분위수 임계값을 **design 기간(≤2022-12)에서만 계산**하고 validation 기간에 적용(또는 rolling/expanding 분위수). 07의 design/validation 분리 취지와 정합.

### 4.7 매직넘버 → config 상수화 + docstring/타입힌트 보강

임계값·가중치·기간 경계·연율화 계수(12, √12)를 `config.py`로. `10_build` 핵심 함수와 plot 함수에 타입힌트·docstring 추가. 전역 `filterwarnings("ignore")`는 범위를 좁히거나 제거.

## 5. 실행 우선순위 로드맵

발표(7/13)가 가깝다는 점을 고려한 현실적 순서입니다.

| 순위 | 작업 | 이유 | 난이도 |
|---|---|---|---|
| 1 | `alignment_flag` 실제 계산 + 날짜 연속성 검증 | 결과 신뢰성 직결, 소규모 수정 | 낮음 |
| 2 | 거래비용 백테스트 반영(4.5) | v3·최종 판단표의 전제, 회의 확정사항 | 중간 |
| 3 | `common/metrics.py`·`common/backtest.py`로 중복 통합(4.1 일부) | 20의 지표 불일치 등 실제 버그 제거 | 중간 |
| 4 | `10_` 넘버링 충돌 정리(4.2) | 재실행 안정성 | 낮음 |
| 5 | yfinance→CSV 기본화 + 스냅샷 저장(4.3) | 결과 재현성 | 중간 |
| 6 | 분류 단계 미래참조 제거(4.6) | 엄밀성(발표 Q&A 대비) | 중간 |
| 7 | 공통 모듈 전면 리팩터링·테스트 도입(4.1·4.4 전체) | 장기 유지보수, 발표 후 진행 가능 | 높음 |

> 발표 전에는 1~4(신뢰성·재현성 직결)를, 발표 후 정리 단계에서 5~7(구조 개선)을 권장합니다.
