# 팩터의 파이프라인·시각화 반영 설계 (2026-07-03)

팩터(팩터 후보 통합표 기준)를 현재 `src/pipeline`·`src/dashboard`에 반영하는 방법 설계.
핵심 원칙: **두 목적을 분리**한다.

- **정적 팩터 로딩 (설명/attribution)** — 기존 λ=0.1/0.3·baseline·BM 전략이 *무엇에 노출됐는지* 설명. RA 요구사항의 "팩터 분석"을 직접 충족. 전략을 바꾸지 않으므로 안전하고 우선.
- **동적 λ (확장)** — 팩터로 λ를 조절. 전략을 바꾸므로 반드시 train/validation/test·walk-forward 검증. 후속 과제.

이 문서는 **정적 로딩 + attribution + 시각화**를 1차 범위로 한다.

---

## A. 파이프라인 반영

### A.1 위치와 데이터 흐름

팩터 로딩은 백테스트 *이후*의 사후 분석이다. `stage_selection`이 만든 전략별 월수익률·turnover를 입력으로 신규 모듈 `stage_factor.py`를 추가한다.

```
stage_selection (백테스트 시계열)
        │  전략별 월수익률 R_strategy, turnover
        ▼
stage_factor
  1) build_factor_matrix()   팩터별 월값 → expanding/rolling z-score, macro/US 시차
  2) screen_factors()        상관·VIF로 중복 제거
  3) full_period_loading()   (R_strategy − R_BM) ~ 팩터 OLS → β·t-stat·R²
  4) rolling_loading()       36개월 rolling OLS → 로딩 시계열
        ▼
  factor_loading_summary.csv / factor_loading_timeseries.csv
```

### A.2 stage_factor.py 함수 스펙

| 함수 | 입력 | 출력 | 비고 |
|---|---|---|---|
| `build_factor_matrix(factors_raw)` | 팩터 원자료(Date + 팩터열) | 표준화·시차 적용된 팩터 행렬 | expanding z-score(룩어헤드 차단), macro/US lag=1 |
| `screen_factors(X)` | 팩터 행렬 | 상관행렬·VIF·채택 목록 | 팀검토 반영(VKOSPI↔실현변동성, term spread↔채권 중복 제거) |
| `full_period_loading(y_excess, X)` | 초과수익·팩터 | β, t-stat, R² | numpy lstsq 또는 statsmodels OLS |
| `rolling_loading(y_excess, X, window=36)` | 초과수익·팩터 | 로딩 시계열 | 24개월은 부록 |

**설계 포인트**: stage_factor는 "주어진 팩터 컬럼 위에서 동작"하도록 만든다. 팩터 데이터가 아직 없어도 로직은 **합성 데이터 단위 테스트**로 검증 가능.


### A.4 stage_attribution.py (부수)

BM 대비 초과수익을 단순 분해한다.

```
초과수익 = 자산배분효과(SAA: w_strategy − w_BM) + 타이밍효과(HSI overlay) + λ smoothing효과 + 비용효과
```

- 자산배분효과: Σ (w_strategy,t − w_BM,t) · r_asset,t
- 타이밍/λ 효과: baseline(즉시비중) 대비 λ 실현비중 차이의 수익 기여
- 비용효과: turnover × cost_rate

## C. 반영 순서 (권장)

1. **0단계 데이터 인제스트**: ECOS/KRX → `data/factors/monthly_factors.csv` (최대 실작업)
2. **stage_factor 정적 로딩** → 요약표 + 히트맵 (RA 팩터 분석 갭을 가장 직접 메움)

5. **(별도) 동적 λ** — train/validation/test 하에서만

## D. 발표 정합성

- 정적 로딩·attribution = "기존 λ가 실제로 무엇에 노출됐는지 설명" → optimizer 답변("최적화 안 함")과 모순 없음.
- 동적 λ = 반드시 "후속 과제 + OOS 검증 전제"로만 언급 → 방어 논리 일관성 유지.

---

## E. 1차 구현 대상 (이번 착수)

`stage_factor.py`를 **"주어진 팩터 매트릭스 위에서 동작하는 정적 + rolling 로딩"** 으로 구현하고,
`common.metrics`/numpy 기반으로 회귀·표준화·VIF를 넣어 **합성 데이터 단위 테스트**로 검증한다.
(팩터 데이터 인제스트 전에도 로직 검증 가능하도록 설계)

필요 의존성: numpy(있음), statsmodels(t-stat용, 없으면 numpy lstsq + 수동 t-stat로 대체).
