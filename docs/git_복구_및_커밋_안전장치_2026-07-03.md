# Git 복구 및 커밋 안전장치 안내 (2026-07-03)

리팩터링을 재개하기 전에 반드시 읽어주세요. 현재 git 상태와, 작업이 다시 사라지지 않게 하는 방법을 정리했습니다.

---

## 1. 무슨 일이 있었나

- 지난 세션에서 만든 `src/common`·`src/pipeline`·`src/tests` 코드와 legacy 이동이 **merge/reset 과정에서 사라졌습니다**(현재 빈 폴더 + `__pycache__`만 남음).
- 원인: **이 작업 환경(샌드박스)에서는 저장소의 `.git` 폴더에 파일을 새로 만들 수는 있어도 기존 파일을 삭제·덮어쓸 수 없습니다.** git 커밋은 index와 ref를 덮어써야 하므로 여기서는 커밋이 완결되지 못했고, 커밋되지 않은 작업이 merge 때 유실됐습니다.
- 그 결과 `.git/index`가 손상되어(`unknown index entry format`) 이 환경에서는 `git status`/`git log`도 실패합니다.

## 2. 중요한 사실: 데이터는 안전함

브랜치와 커밋 자체는 살아 있습니다(참조용):

| 항목 | 값 |
|---|---|
| 현재 브랜치(HEAD) | `codex/시각화_문서_정리` |
| 해당 브랜치 커밋 | `10f5344d700f2a518d61cb8ff6d4b1bdd9e1150e` (merge origin/parameterized_code_기초실험) |
| main | `5a3543335dfc8533231b2d5d0894e73d83022734` |
| ORIG_HEAD | `474c9cb325749b9d4b8528f26174e99e6bf14d1d` |

여기서 보이는 손상은 대부분 **샌드박스에서 마운트를 통해 볼 때의 문제**이며, PC의 실제 git에서는 정상이거나 쉽게 복구됩니다.

## 3. 지금 만들어 둔 안전장치 (git 무관)

- **파일 백업**: `_backups/src_20260703_115214/` 에 현재 `src/` 전체(42개 .py)를 복사해 두었습니다. git 상태와 무관하게 언제든 복구 가능합니다.
- **백업 스크립트**: `backup_src.sh` — 위험한 작업 전에 `bash backup_src.sh` 를 실행하면 `_backups/src_<날짜시간>/` 로 스냅샷을 만듭니다.

## 4. PC(Windows)에서 할 복구 절차

git 작업은 **반드시 PC 네이티브 환경**에서 하세요(샌드박스 아님).

1. 저장소 폴더에서 `git status` 실행.
2. 인덱스 손상 메시지가 나오면 인덱스를 재생성:
   ```powershell
   del .git\index
   git reset            # HEAD(codex/시각화_문서_정리) 기준으로 인덱스 재생성
   git status
   ```
3. 잘못 남은 임시 파일 제거(샌드박스가 못 지운 것):
   ```powershell
   del .git\_write_test
   ```
4. 현재 작업트리(조원 merge 상태 + 새 파일)를 커밋:
   ```powershell
   git add -A
   git commit -m "merge 조원 최종 스크립트 및 리포트 아티팩트"
   ```
5. 만약 브랜치 ref가 깨져 HEAD를 못 찾으면:
   ```powershell
   git symbolic-ref HEAD refs/heads/codex/시각화_문서_정리
   git update-ref refs/heads/codex/시각화_문서_정리 10f5344d
   ```

## 5. 앞으로의 작업 원칙 (재발 방지)

1. **리팩터링 결과는 이 환경에서 커밋되지 않습니다.** 한 덩어리 작업이 끝날 때마다 PC에서 `git add -A && git commit` 로 즉시 커밋하세요.
2. 위험한 작업(merge/reset/대규모 이동) 전에는 `bash backup_src.sh` 로 스냅샷을 먼저 만드세요.
3. 다음 merge를 하기 전에, 현재 작업트리를 반드시 먼저 커밋하세요. 커밋되지 않은 변경은 merge 때 사라질 수 있습니다.
