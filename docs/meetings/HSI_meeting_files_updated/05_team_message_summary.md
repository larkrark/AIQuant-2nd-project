# 05. 팀 공유용 요약 문구

## A. 전체 회의 공유용 요약

현재는 전체 전략 성과를 확정한 단계가 아니라, HSI 입력 데이터 생성 및 분위수/z-score 신호 산출 예비 실행 결과를 확인한 단계입니다.

예비 ETF 3종을 대상으로 일별 가격, 월말 가격, 월간 수익률, 분위수 기반 HSI 결과, z-score 기반 HSI 결과가 CSV로 저장되는 것을 확인했습니다.

다만 현재 결과는 전체 ETF 유니버스 기준이 아니라 3개 ETF 예비 유니버스 기준이며, HSI 신호는 일별, 백테스트 수익률은 월별로 계산되어 있습니다. 따라서 이후에는 월말 HSI 상태를 추출해 다음 달 월간 수익률에 적용하는 방식으로 시점 기준을 맞추는 것이 필요합니다.

점수화 방식은 분위수 방식을 기본값으로 두고, z-score는 삭제하지 않고 보조 비교 실험으로 유지하는 방향을 제안합니다.

추가로, 현재 코드에서 사용되는 `buy / watch / caution` 라벨은 계산 결과를 빠르게 확인하기 위한 1차 라벨로 두고, 보고서와 정식 1차 테스트에서는 HSI의 원래 설계 의도를 살려 별도의 상태 정의표로 재해석하는 방향을 제안합니다.

## B. 데이터 담당자에게 전달할 내용

공통 입력 신호는 동일하게 유지하고, HSI 점수화 방식만 분위수 방식과 z-score 방식으로 비교하면 좋겠습니다.

다만 이전 HSI 예비 실험에서 분위수 방식으로 HSI를 구현해 본 흐름이 있기 때문에, 이번 팀 프로젝트의 초기 실험 기준도 분위수 방식으로 이어가면 실험 흐름이 자연스럽다고 봅니다.

z-score 방식은 삭제하지 않고, 같은 입력자료를 다른 방식으로 표준화했을 때 결과가 어떻게 달라지는지 확인하는 보조 비교 실험으로 남기면 좋겠습니다.

정리하면, 본 실험의 기본 점수화 기준은 분위수 방식으로 두고, z-score 방식은 비교용 보조 실험으로 유지하는 방향을 제안합니다.

추가로 현재 HSI 결과는 일별 신호이므로, 월별 백테스트에 연결하기 위해서는 월말 HSI 상태를 별도로 추출하는 단계가 필요합니다.

또한 현재 코드의 `buy / watch / caution` 라벨은 계산 과정에서 결과를 빠르게 확인하는 데 유용하므로 1차 라벨로 유지하고, 보고서용 상태명은 별도 상태 정의표에서 정리하면 좋겠습니다. 이렇게 하면 기존 계산 흐름은 유지하면서도, HSI가 단순 매매 신호가 아니라 시장상태 해석용 보조지표라는 점을 더 분명하게 설명할 수 있습니다.

## C. 회의록 담당자에게 전달할 내용

이번 회의 합의사항으로는 “HSI 기본 실험의 점수화 기준은 분위수 방식으로 두고, z-score 방식은 비교용 보조 실험으로 유지한다”는 내용을 기록하면 좋겠습니다.

근거는 이전 HSI 예비 실험에서 분위수 방식으로 HSI를 구현해 본 흐름이 있고, 분위수 방식이 서로 단위가 다른 입력 신호를 과거 분포 내 상대적 위치 기준으로 비교하기에 안정적이라는 점입니다.

또한 z-score 방식은 평균과 표준편차를 기준으로 신호가 얼마나 크게 벗어났는지 확인할 수 있으므로, 위기 구간이나 급격한 변동 구간에서 보조 비교 기준으로 활용할 수 있습니다. 따라서 z-score는 삭제하지 않고 비교 실험으로 남깁니다.

현재 결과는 전체 전략 성과 검증 결과가 아니라, 예비 ETF 3종 기준으로 HSI 입력 데이터 생성과 분위수/z-score 신호 산출 흐름을 확인한 예비 실행 결과입니다.

추가로 일별 HSI 신호를 월별 백테스트에 연결하기 위해 월말 HSI 상태를 추출하고, 다음 달 ETF 월간 수익률에 적용하는 방식으로 시점 기준을 맞추는 것이 필요합니다.

상태명 관련 합의사항으로는, 현재 코드의 `buy / watch / caution` 라벨은 1차 계산 라벨로 유지하되, 보고서와 정식 1차 테스트에서는 위험 완화, 관찰/중립, 위험 악화, 필요 시 충돌 상태와 같은 프로젝트용 HSI 상태명으로 매핑하는 별도 상태 정의표를 둔다는 내용을 기록하면 좋겠습니다.

## D. 발표자에게 전달할 내용

발표에서는 z-score와 분위수를 단순한 코드 옵션이 아니라, 서로 단위가 다른 입력 신호를 같은 기준으로 맞추는 스케일링 방식으로 설명하면 좋겠습니다.

본 실험은 이전 HSI 예비 실험과의 연속성을 고려해 분위수 방식을 기본 기준으로 사용하고, z-score는 보조 비교 실험으로 남겨 두는 구조입니다.

이렇게 설명하면 “왜 분위수를 기본값으로 선택했는가”와 “왜 z-score를 삭제하지 않았는가”를 모두 자연스럽게 설명할 수 있습니다.

다만 발표에서는 z-score와 분위수 비교가 프로젝트의 본 주제가 아니라, HSI 기반 ETF 방어형 자산배분 전략을 안정적으로 구현하기 위한 보조 비교 실험이라는 점을 분명히 하는 것이 좋습니다.

또한 현재 결과는 전략 성과 검증 결과가 아니라, HSI 신호 산출 예비 실행 결과입니다. 이후 EW+HSI 백테스트부터 시작해 RP, GMV, MDP로 확장하는 순서로 설명하면 흐름이 자연스럽습니다.

HSI 상태명 설명에서는 `buy / watch / caution`을 최종 매매 신호처럼 설명하기보다, 계산 결과 확인용 1차 라벨이라고 설명하는 편이 안전합니다. 발표 본문에서는 HSI의 원래 의도에 맞게 위험 악화 신호와 위험 완화 신호의 방향, 강도, 충돌 정도를 해석하는 시장상태 분류 체계로 설명하면 좋겠습니다.

## E. HSI 상태명 정식화 관련 공유 문구

현재 코드에서 생성되는 `buy / watch / caution` 라벨은 계산 결과를 빠르게 확인하기 위한 1차 라벨로 유지하되, 보고서와 정식 1차 테스트에서는 HSI의 원래 해석 구조를 살리기 위해 별도의 상태 정의표를 두는 방향을 제안합니다.

HSI는 단순히 매수·관망·주의를 판단하는 신호라기보다, 위험 악화 신호와 위험 완화 신호의 방향, 강도, 충돌 정도를 함께 해석하는 시장상태 분류 체계입니다. 따라서 기존 라벨을 바로 삭제하거나 바꾸기보다는, `buy / watch / caution`을 프로젝트용 HSI 상태명으로 매핑하는 방식이 적절해 보입니다.

예를 들어 `buy`는 위험 완화 우세, `watch`는 관찰 또는 중립, `caution`은 위험 악화 우세로 해석할 수 있습니다. 다만 위험 악화와 위험 완화 신호가 동시에 강하게 나타나는 경우에는 `conflict` 또는 `mixed` 상태를 별도로 둘지 회의에서 확인하면 좋겠습니다.

## F. 김근형 담당 진행상황

김근형은 공유받은 HSI 신호 계산 모듈을 Spyder 및 VS Code 환경에서 실행해 보고, ETF 후보 선정, 가격 데이터 로드, 기본 HSI 계산, 예비 Grid Search 단계가 작동하는지 확인하였다.

실행 과정에서 괄호 누락, note 출력 방식, 일부 기본 설정값 등 실행에 필요한 부분만 최소 수정하였고, 기존 코드의 큰 구조는 유지하였다.

또한 이전 HSI 예비 실험에서 분위수 방식으로 HSI를 구현한 흐름을 바탕으로, 이번 팀 프로젝트의 초기 실험에서도 분위수 방식을 기본 점수화 기준으로 두고, z-score 방식은 삭제하지 않고 비교용 보조 실험으로 남기는 방향을 제안하였다.

현재 예비 ETF 3종 기준으로 일별 가격, 월말 가격, 월간 수익률, 분위수 방식 HSI 결과, z-score 방식 HSI 결과가 저장되는 것을 확인하였다.

다만 현재 결과는 전체 전략 성과 검증 결과가 아니라, HSI 입력 데이터 생성 및 신호 산출 예비 실행 결과이다.

현재 Grid Search는 최종 전략 선정용이 아니라, 여러 파라미터 조합을 반복 실행할 수 있는지 확인하는 예비 점검용 단계이다. 최종 Grid Search는 이후 백테스트 성과지표인 CAGR, MDD, Sharpe, Turnover 등을 연결한 뒤 진행할 예정이다.

이후 작업에서는 일별 HSI 신호에서 월말 HSI 상태를 추출하고, 이를 다음 달 ETF 월간 수익률에 적용하는 방식으로 EW+HSI 백테스트부터 연결할 예정이다.

추가로, 이전 HSI 예비 실험에서 사용했던 위험 악화·위험 완화·충돌 중심의 해석 체계를 현재 프로젝트에서도 상태 정의표로 복구하고, `buy / watch / caution` 라벨과 연결하는 방식을 제안하였다.
