## 주요 내용

### intensity 역할 재정의 (김근형)
- intensity는 상태분류 전체의 주조건이 아니라 **conflict 판별과 사후 진단용 보조지표**로 재정의
- 근거: accident_zone에서는 정의상 |direction| ≤ intensity가 항상 성립 → risk_component 중심으로 단순화가 더 해석 가능. 반면 conflict 구간에서는 의미 있는 보조 조건으로 작동한 사례 있음(2026-06)
- 모래시계·3차원 구조는 산술 모델이 아닌 개념적 시각화임을 분리 표기, 오해 소지가 크면 부록으로 이동

### 강사님 예상질문 답변 정리 (김근형)
1. **"결국 최적화이지 않은가?"** → 그리드 위 후보 비교라 최적화적 성격은 있으나, 목적함수(score = a×Calmar + b×Sharpe − c×Turnover 등)를 최대화한 것이 아님. 사전 정의 λ 후보를 OOS 10bp 비용차감 기준으로 비열등 판정. 표현은 "목적함수 최대화형 최적화가 아니라 **그리드 기반 후보 선별**"
2. **"그냥 라벨링한 것 아니냐?"** → 상태 이름을 붙인 건 맞지만 머신러닝식 정답 라벨링 아님. 미래 결과를 보고 라벨을 만든 것이 아니라 **사전에 정한 계산 규칙**(V+, V−, direction, intensity, conflict 기준)으로 분류. 답변 구분표 그림 제작

### 입력변수 선정 근거 (김민호, 발표 슬라이드용)
- ①수익률(return, 20일) — 단기 방향 즉각 감지 ②이평 대비 위치(ma_pos, 20·60·120일) — 추세 이탈 ③모멘텀(21·63·126일) — 추세의 힘·지속성 ④변동성(vol, 20일) — 방향과 무관한 유일한 '위험' 축 ⑤상대강도(rs, 21일) — risk-off 로테이션 포착
- 팀 이견 없음 → 슬라이드화

### conflict 판정 로직: 문서 vs 코드 불일치 확인 — 핵심 논의
- 김민호: "conflict는 왜 안 쓰고 폐기했는지" 질문 → 권보성 확인 결과 **폐기 아님**: 감지 비율 2.8%로 낮지만 상태로 사용됨
- 권보성 코드 확인:
  - `config.py`의 STATE_TARGET_WEIGHTS에 conflict 목표비중 [0.35, 0.40, 0.25] 반영
  - 최종 구현은 conflict ratio 수식(2×min(V+,V−)/(V+,V−))으로 직접 분류하지 않고, ①ETF별 buy·caution 동시 발생 ②주 위험자산 |direction| ≤ 0.15 & intensity ≥ high_intensity_cutoff(표본 내 75% 분위수, zscore 0.383981/rank 0.565476)의 두 경우를 conflict로 분류
  - 최종보고서 기준 conflict 4개월 발생(전체 172개월 중 2.33%, 유효 168개월 중 2.38%)
- 김근형: config.py는 정의가 아니라 판정 후 적용할 목표비중표. 실제 판정 로직은 05/04/16/33번 파일에서 확인. 코드 버전별 용어 차이 정리:
  - 05번(1차 실험): risk_score/overheat_score/recovery_score
  - 16번(state5 재분류): direction/intensity/cross_asset_conflict
  - 33번(후속 state5): hsi_direction/hsi_intensity/conflict_ratio
- 김민호: "문서의 판정 기준이 실제 로직과 다른 것 확인 — 오히려 HSI 원래 컨셉과 부합해 다행". 남은 부분은 주말에 마무리
